# /adopt-dev-env

Infer this repository's **local development environment** from its codebase and generate:

- `.cursor/local/bin/config.sh` ‚Äî project-specific runtime config for the dev scripts
- `docs/ai/dev-runtime.md` ‚Äî human + agent reference for local dev
- `.cursor/commands/db-migrate.md`, `db-rollback.md`, `db-seed.md` ‚Äî when a migration tool is detected

This command works across stacks (Node.js, Python, Go, mixed). The agent infers freely ‚Äî nothing is hardcoded. Re-run after major infrastructure changes.

## Prerequisites

1. `cursor-kit init` already applied (`.cursor/` is a real directory with managed kit).
2. Run `/adopt-repo-docs` first if `docs/ai/` does not exist ‚Äî that command creates the `docs/ai/` scaffold; `/adopt-dev-env` only adds or updates `dev-runtime.md` within it.
3. Delegate to **`devops-agent`** as primary owner for script generation; **`docs-agent`** for `dev-runtime.md`.

## Required workflow

### 1. Discover (read-only)

Scan for dev environment signals. Read **only what you need** ‚Äî do not load every file:

**Process and script signals**
- `package.json` ‚Üí `scripts` block: look for `dev`, `start`, `build`, `test`, `watch`, and any worker/scheduler/queue scripts. Note package manager (`pnpm-workspace.yaml`, `yarn.lock`, `.npmrc`).
- `Procfile` ‚Üí process list (Heroku/Foreman style).
- `Makefile` ‚Üí targets named `dev`, `up`, `down`, `start`, `stop`, `run`, `serve`.
- `docker-compose.yml` / `compose.yaml` / `docker-compose.*.yml` ‚Üí services, `command:`, ports, `profiles:`.

**Infrastructure signals**
- Docker Compose ‚Üí compose file path(s), services, `depends_on`, healthchecks.
- `Makefile` ‚Üí infra targets (`infra-up`, `infra-down`, `db-up`, etc.).
- No Docker ‚Üí check if the repo is standalone (all deps in process, bare `npm run dev`).

**Database / migration signals** (if any of these exist, generate DB commands)
- `prisma/schema.prisma` ‚Üí Prisma (`npx prisma migrate dev`, `npx prisma db seed`)
- `drizzle.config.ts` or `drizzle.config.js` ‚Üí Drizzle (`npx drizzle-kit migrate`)
- `alembic.ini` ‚Üí Alembic Python (`alembic upgrade head`, `alembic downgrade -1`)
- `db/migrate/` or `database/migrate/` ‚Üí Rails-style (`rails db:migrate`, `rails db:rollback`)
- `migrations/` + `go.mod` ‚Üí Go migrate or golang-migrate
- `flyway.conf` ‚Üí Flyway (`flyway migrate`, `flyway undo`)
- `liquibase.properties` ‚Üí Liquibase (`liquibase update`, `liquibase rollbackCount 1`)
- `package.json` `scripts` with `db:migrate`, `db:rollback`, `migrate`, etc.

**Python / Go / other stack signals**
- `pyproject.toml` / `requirements.txt` / `Pipfile` / `setup.py` ‚Üí Python; note virtual env tool (`uv`, `poetry`, `pipenv`, `venv`)
- `manage.py` ‚Üí Django; start: `python manage.py runserver`
- `go.mod` ‚Üí Go; start: `go run ./cmd/...` or `air` if present
- `.env.example` / `.env.sample` ‚Üí note env var names (do not read actual `.env`)

### 2. Build the runtime profile

From the scan, determine:

| Field | Derived from |
|-------|-------------|
| `PROJECT_NAME` | repo directory name or `name` in `package.json` / `go.mod` |
| `PROCESSES` | scripts/Procfile/Compose services that must run during local dev |
| `INFRA_UP_CMD` | Compose up / Make target / empty string if no infra |
| `INFRA_DOWN_CMD` | Compose down / Make target / empty string |
| `PORTS` | ports from Compose, well-known defaults (3000, 8000, 8080, 5432, 6379‚Ä¶) |

**Process selection heuristic:**
- Start with the primary app process (web server / API).
- Add background workers only if they have an explicit script or Procfile entry.
- Do **not** add every Compose service as a process ‚Äî Compose services are infra, not `PROCESSES`.
- Maximum 6 processes; prefer fewer and let the user add more manually.

### 3. Generate `.cursor/local/bin/config.sh`

Overwrite (not append) the file with inferred values. The file format **must** match the array format:

```bash
#!/usr/bin/env bash
# Generated by /adopt-dev-env ‚Äî re-run to regenerate from repo inspection.

PROJECT_NAME="<inferred-name>"
DEV_ROOT="/tmp/${PROJECT_NAME}-dev"
LOG_DIR="$DEV_ROOT/logs"
RUN_DIR="$DEV_ROOT/run"

PROCESSES=(
  "app|<start-command>"
  # "worker|<worker-command>"   # uncomment if repo has a worker
)

INFRA_UP_CMD="<compose-up or make target or empty>"
INFRA_DOWN_CMD="<compose-down or make target or empty>"

PORTS=(
  "app|<port>"
  # "postgres|5432"
)
```

**Rules:**
- `name|command` ‚Äî the pipe `|` separates name from command; no spaces around it.
- Empty `INFRA_UP_CMD=""` / `INFRA_DOWN_CMD=""` is valid (scripts guard with `[[ -n ... ]]`).
- Comments are encouraged ‚Äî leave commented-out entries for optional processes the user may want.

### 4. Update `docs/ai/dev-runtime.md`

Create or overwrite with **`docs-agent`**. Required sections:

```markdown
# Dev Runtime

## Quick start
\`\`\`bash
bash .cursor/local/bin/up.sh    # start all processes
bash .cursor/local/bin/status.sh
bash .cursor/local/bin/down.sh
\`\`\`

## Processes
| Name | Command | Port |
...

## Infrastructure
How infra is managed (Compose file, services, healthchecks).

## Environment variables
Required `.env` keys (names only, no values). Link to `.env.example` if present.

## Database
Migration tool, seed command, rollback command (if applicable).

## Useful scripts / make targets
Any additional dev commands worth knowing.
```

### 5. Generate DB commands (when migration tool detected)

If any migration signal was found, create these command files in `.cursor/commands/`:

**`.cursor/commands/db-migrate.md`**
```markdown
# /db-migrate

Apply pending database migrations to the local dev database.

## Command
<exact shell command to run migrations>

## Notes
- Requires infra running (`/dev-up` first, or `bash .cursor/local/bin/up.sh`).
- Tool: <migration tool name and config file>.
- On failure: check `<log location>` or run `<status command>`.
```

**`.cursor/commands/db-rollback.md`**
```markdown
# /db-rollback

Roll back the most recent database migration.

## Command
<exact shell command>

## Notes
- Rolls back one step by default. <tool-specific flag for multiple steps>.
- Tool: <migration tool name>.
```

**`.cursor/commands/db-seed.md`**
```markdown
# /db-seed

Seed the local dev database with baseline data.

## Command
<exact shell command, or "No seed script found ‚Äî add one here.">

## Notes
- Idempotent: <yes/no/depends ‚Äî describe>.
- Data location: <seed file paths if known>.
- Use `seed-agent` to generate seed fixtures for new entities.
```

These command files are **not** managed by cursor-kit (they don't exist in cursor-base's shared `.cursor/`). Cursor-kit update will not overwrite them.

### 6. Verify scripts are consistent

After writing `config.sh`, mentally verify:
- Each `PROCESSES` entry's command exists in the repo (`package.json` script, binary, or executable).
- `INFRA_UP_CMD` references a Compose file that actually exists (or is empty).
- Every `PORTS` port is used by at least one process or infra service.

If any entry cannot be inferred confidently, leave it as a commented placeholder and note the gap in the output report.

## Output (required)

- **Runtime profile** ‚Äî project name, stack, processes, infra approach (1 table or bullet list).
- **Files written** ‚Äî path and one-line summary for each file created or updated.
- **DB detection** ‚Äî which tool was detected and which commands were generated (or "no DB migration tool detected").
- **Gaps / follow-ups** ‚Äî entries left as placeholders, manual steps needed, recommended next commands.
- **Next step hint** ‚Äî if UI stack exists and `/adopt-design-system` hasn't been run, mention it.
